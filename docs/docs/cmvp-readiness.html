CMVP readiness — cryptographic module preparation

This document is a preparation skeleton for a future FIPS 140-3 / CMVP submission. It defines the cryptographic module boundary, the approved algorithm set, required self-tests, and a submission checklist. It does not assert validation — that is issued by an accredited CST lab + NIST/CSE after testing. Use it to scope the work and to structure the eventual Security Policy (per SP 800-140B / ISO/IEC 19790 + 24759).

1. Module definition

  • Module name (proposed): robobus FIPS Crypto Sidecar
  • Type: software cryptographic module (Level 1 target).
  • Operational environment: the validated backend's tested platform (see §4) running the sidecar process (robobus.crypto.sidecar).
  • Embodiment: the sidecar process is the module; applications are outside the boundary and call it over the local AF_UNIX socket. Keeping the boundary in a separate process is deliberate — it isolates keys and the approved algorithm implementations from application code.

2. Cryptographic boundary

  ┌──────────────────────── application (OUTSIDE boundary) ────────────────────────┐
  │  robobus bus / ROS / LSL adapters  ──(AF_UNIX: seal/open requests)──►           │
  └──────────────────────────────────────────────────┬─────────────────────────────┘
                                                      ▼
  ┌──────────────────── robobus FIPS Sidecar (MODULE BOUNDARY) ─────────────────────┐
  │  request handler  →  AEAD (AES-256-GCM)  ──►  validated crypto backend (§4)      │
  │  key storage (in-memory, zeroized on exit)   self-tests (§5)   role/service (§3) │
  └──────────────────────────────────────────────────────────────────────────────────┘

The application never holds the AEAD key; only ciphertext crosses the boundary. KEM / signature operations for the handshake can likewise be moved inside the boundary by extending the sidecar protocol (ops reserved for ML-KEM encaps/decaps and ML-DSA sign/verify).

3. Roles, services, authentication (Level 1)

Role Services
Crypto Officer configure backend, load/rotate key, start/stop module, run self-tests
User seal (encrypt+auth), open (decrypt+verify)

Level 1 requires no operator authentication. Services map 1:1 to sidecar protocol ops.

4. Approved algorithms (CAVP requirement)

Each algorithm needs a CAVP (Cryptographic Algorithm Validation Program) certificate from the validated backend. robobus uses approved algorithms; the certificates come from the backend you bind (OpenSSL 3 FIPS provider / AWS-LC-FIPS / PKCS#11 HSM).

Function Algorithm Standard CAVP cert
AEAD AES-256-GCM FIPS 197 / SP 800-38D (from backend)
KEM ML-KEM-512/768/1024 FIPS 203 (from backend)
Signature ML-DSA-44/65/87 FIPS 204 (from backend)
Signature SLH-DSA FIPS 205 (from backend)
Hash SHA-256/384/512 FIPS 180-4 (from backend)
XOF/Hash SHA3, SHAKE FIPS 202 (from backend)
KDF HKDF SP 800-56C / 800-108 (from backend)
DRBG CTR/Hash_DRBG SP 800-90A (from OS/backend)

Non-approved (must be disabled in FIPS mode — enforced by compliance.POLICY['fips']): ChaCha20-Poly1305, X25519/X448 ECDH.

5. Self-tests (implemented)

robobus.crypto.compliance.run_self_tests() runs at start-up and on demand:

  • KATs — AES-256-GCM (NIST vector), SHA-256, SHA-384, SHA3-256, SHA3-512.
  • Pairwise-consistency tests — ML-KEM-512/768/1024, ML-DSA-44/65/87, SLH-DSA.
  • On any failure the module must enter an error state and refuse service (the policy checker returns a non-zero exit and refuses suite selection).

To match CMVP exactly, add: software integrity test (HMAC/signature over the module binary at load), and deterministic ML-KEM/ML-DSA KATs using the FIPS test seeds (the backend's CAVP harness provides these).

6. Key management

  • Keys are 256-bit, generated by an SP 800-90A DRBG (secrets.token_bytes) or supplied via key file (mode 0600) / ROBOBUS_KEY.
  • Session keys derive from the KEM shared secret via HKDF, bound to the handshake transcript.
  • Keys live only in module process memory and are released on exit. For Level 2+ add explicit zeroization and a key-storage IG mapping.

7. Submission checklist

  • [ ] Choose and bind a CMVP-validated backend (OpenSSL 3 FIPS / AWS-LC-FIPS / HSM).
  • [ ] Pin the tested operational environment (OS + version + CPU).
  • [ ] Collect CAVP certificates for every algorithm in §4.
  • [ ] Add module software-integrity self-test at load.
  • [ ] Add deterministic KATs with FIPS seeds for ML-KEM/ML-DSA.
  • [ ] Write the full Security Policy (SP 800-140B template; expand §§1–6 here).
  • [ ] Engage an accredited CST laboratory for testing.
  • [ ] Document key zeroization + finite state model.

8. Related standards

ISO/IEC 19790 (≈ FIPS 140-3) and ISO/IEC 24759 (test requirements) mirror this structure internationally; CNSA 2.0 (see COMPLIANCE.md) constrains the algorithm choices to Level 5 (cnsa20 suite). CSfC requires two validated layers — model with the csfc suite plus a second validated transport (e.g., MACsec/IPsec) beneath.
