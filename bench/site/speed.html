requirements · speed under each profile

Pick your requirement, see the cost

Security is a spectrum you choose per link, not a mandate — from classical (backwards-compatible with non-PQ peers) through hybrid transition profiles to CNSA 2.0 for National Security Systems. Each profile splits into two cost centres: session setup (the ML-KEM / ML-DSA handshake, once per peer, amortized) and the per-message hot path (AES-256-GCM, every frame). Session cost swings ~130× with assurance; per-message stays flat at line rate — so higher security buys handshake latency, not throughput.

crypto · every primitive · every language · grouped by technique

The crypto matrix

robobus is a post-quantum bus, so its crypto is a whole suite, not one cipher. Here is every primitive it uses, measured in every language that can do it natively — grouped per technique so each bar chart is a coherent apples-to-apples comparison (identical workload, languages ranked fastest first). The AEAD and hash groups are each language's native-maximum stack (stdlib where it exists, else the platform OpenSSL through the language's FFI). The post-quantum groups have essentially no native per-language implementations, so they show the real distinct backends instead — OpenSSL 3.6.3 EVP, liboqs, and Go's native pure-Go crypto/mlkem — a genuine implementation comparison; every language reaches the OpenSSL/liboqs numbers via FFI at ~the same cost (see the FFI-bindings section).

python runtimes · CPython · Cython · PyPy+cffi

Same code, four runtimes — pick your stack

robobus's Python surface runs on whatever runtime fits, and each has a native acceleration path. Cython stacks with CPython (seal/open compiled to C in-process); PyPy does not stack with Cython (its cpyext C-API layer is slow) but stacks with native code via cffi — the JIT optimizes straight through the FFI into librobobus. one C-ABI · every language's FFI

Full robobus from any language — over its own FFI

You don't need a native robobus port to get the real thing. librobobus is one C-ABI shared library, and every language reaches its complete seal/open surface through its own foreign-function interface — LuaJIT's native FFI, Perl's FFI::Platypus (libffi), Octave's compiled .oct C-linkage (it has no raw loadlibrary), and CPython's cffi. The same RBX1 seal(41 B)→open round trip (AES-256-GCM), each verified to round-trip before timing. Because the work happens in the shared C, every binding lands near the C ceiling — the FFI overhead is nearly free:

transport × language

The full product

Each cell: that language's native client pushing the sealed frame over that transport (frames/s, loopback, pipelined). These cells are I/O-syscall-bound — every language issues the same send/recv into the same kernel, so throughput converges to the loopback ceiling and small differences (even Python edging C) are run-to-run noise, not language speed. That's the point: the transport sets the ceiling, not the caller — the opposite of the codec table above, where language speed spans ~1,700×.
